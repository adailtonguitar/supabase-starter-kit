## Achados concretos (com números reais do banco)

1. **Core operacional já tem os índices certos.** A migration de

   - `idx_products_company_name` (2.950 scans)
   - `idx_sales_company_created` (25.795 scans)
   - `idx_sale_items_sale` (38.039 scans)
   - `idx_financial_entries_company_due` (683 scans)
   - `idx_clients_company_name` (1.452 scans)
   - `idx_stock_movements_company_created` (228 scans)
   - `idx_company_users_active` (39.285 scans)

2. **Padrão multi-tenant está indexado:** `idx_products_company` tem

3. **Cache hit 100%** — RAM suficiente para todo o workload atual.

4. **zero seq scans** nos índices listados — todos atendem.

### 🔴 Problemas reais identificados

#### [CRÍTICO-1] `user_roles_user_id_role_key` — 30.180.385 scans em 57 dias

Esse é o índice **mais consultado** do banco — ~6 scans por segundo.
É quase certamente a function `is_super_admin()` / `has_role()` sendo
chamada em toda policy/RPC. Não é problema de dados, é de arquitetura:
**cada request do app re-executa esta checagem**.

**Impacto em escala:** a 100 req/s isso vira ~600 lookups/s só pra
verificar role. Quando for 10k empresas, isso explode.

**Sugestão:** cachear role do usuário no JWT (`raw_user_meta_data.role`)
ou em função `STABLE` cached. Se o role do usuário não muda por request,
não precisa ser consultado múltiplas vezes.

#### [CRÍTICO-2] `idx_company_users_company` — 3.898.852 scans

Confirma empiricamente o que foi apontado na auditoria estática: **a
subquery `company_id IN (SELECT company_id FROM company_users WHERE
user_id = auth.uid())` é reavaliada por linha** em dezenas de policies.

**Impacto em escala:** a cada query em `sales`, `products`,
`financial_entries`, etc., este índice é usado. A 10k empresas × 1k
req/s você está dobrando o custo de toda leitura.

**Correção:** função `public.current_user_company_ids() STABLE` (já
preparada na migration `20260421160000`) + reescrever as policies
das tabelas quentes para usar `company_id = ANY (current_user_company_ids())`.

#### [ALTO-1] `profiles` RLS aberto — `USING (true)`

Confirmado na migration `20260409173424_*.sql#L14`. Qualquer usuário
autenticado vê nome/avatar de todo mundo. Violação LGPD se virar SaaS
aberto. **Correção já pronta** em `20260421160000_db_hardening_scale.sql`.

#### [ALTO-2] Duplicação de índices UNIQUE em `products`

Achei dois índices UNIQUE sobre `(company_id, sku)`:
- `products_sku_company_unique` (560 kB, 1 scan em 57 dias)
- `products_company_id_sku_key` (520 kB, 4.154 scans)

Só um é necessário. O primeiro pode ser removido (ocupa 560 kB e
custa em cada INSERT/UPDATE).

```sql
DROP INDEX IF EXISTS public.products_sku_company_unique;
```

#### [ALTO-3] Índice `idx_products_name` (600 kB, 4 scans)

Quase não usado — provavelmente resíduo antigo. `idx_products_company_name`
já cobre o caso de busca por empresa + nome.

```sql
DROP INDEX IF EXISTS public.idx_products_name;
```
